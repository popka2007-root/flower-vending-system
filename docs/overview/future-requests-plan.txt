Назначение
----------
Этот файл - практичный backlog для следующих запросов к Codex/GitHub. Он
сохраняет план cleanup, документации, scripts и product-readiness без
необходимости поднимать историю чата.

Текущий baseline
----------------
- Проект уже является сильной simulator-first control platform.
- Cleanup PR убрал tracked generated/legacy docs, пустые placeholder packages и
  пустой tests fixture package.
- Repository hygiene checks подключены в scripts/verify_project.py.
- Verifier проверяет матрицу конфигов и smoke --help для CLI-команд.
- simulator-ui поддерживает --reset-state для очистки demo SQLite state.
- Docs разложены по docs/overview, docs/operations, docs/architecture,
  docs/hardware, docs/adr и docs/ru; mapping старых путей описан в docs/README.md.
- Hardware target documentation отделена от generic examples и лежит в
  config/targets/.
- Проект пока не production-hardware-ready: real hardware нужно подтверждать на
  bench stand.

Что делать дальше простыми словами
----------------------------------
1. Сначала убрать тонкие .bat/.sh wrappers, которые просто повторяют
   python -m flower_vending.
2. После этого проверить sitecustomize.py и root flower_vending shim: оставить
   только если они реально нужны для запуска.
3. Затем добавить operator diagnostics: status --json и events --limit N.
4. Потом улучшать packaging/release и CI matrix.
5. Реальное железо трогать только после bench inventory и bench validation.

---------------------------------------

Действия:
- Запустить Windows inventory script до изменения OS.
- Запустить Linux inventory script с live USB или установленного Debian target.
- Составить device map для validator, payout, motors, delivery window, pickup
  sensor, door sensor, temperature/cooling и watchdog.
- Снимать DBV-300-SD traces только из explicit operator-provided raw bytes.
- Добавлять production adapters по одному устройству и только за явными config

Нельзя делать без bench hardware:
- Реализовывать production DBV protocol.
- Включать real cash acceptance.
- Реализовывать реальные payout/motor/pickup sensor adapters.
- Заявлять production-ready watchdog, service или autostart behavior.

Готовый промт:

  подготовь bench validation checklist для DBV-300-SD, payout, motors, sensors,
